# КИМ-2.1. [Лабораторная работа]

 Состоит из двух частей:

## Внесение изменений в репозиторий для улучшения его качества.

Внесение изменений может включать по отдельности или вместе следующие изменения: 
1. копирование коммитов между ветками, 
2. удаление изменений, 
3. линеаризацию репозитория, 
4. идеализацию истории: 
    1. перемещение коммитов (в том числе через конфликтующие коммиты),
	2. слияние коммитов
	3. разбиение коммитов.
	
При этом запланированы типовые конфликты при выполнении действий и студенты должны их разрешить.

Базовые репозитории для этой части разрабатывают студенты на основе требований (M3-advanced-git/kim-02-requirements.md), при этом при выполнении студенты обмениваются репозиториями и выполняют работу по репозиторию другого человека.

Критерии оценивания:
* корректность выбора комманд Git и определения их параметров - 0-2 балла
* корректность разрешения конфликтов - 0-2 балла

Данная часть лабораторной работы проводится в два этапа:
1. Подготовка репозиториев и тренировка. Студенты во время самостоятельной работы готовят тренировочные репозитории в соответствии с требованиями для выполнения различных изменений и тренируются на них.
2. На лабораторной работе студенты обмениваются репозиториями (выбранными с помощью генератора случайных чисел) и выполняют задания на репозиториях друг друга.


## Ролевая игра по совместной разработке программного кода. 

Группа студентов играет роль команды разработчиков, которой необходимо внести изменения в большой репозиторий работая одновременно. Изменения должны вноситься как в главную ветку разработки (dev, main, master) симулируя срочные багфиксы работающего кода, так и в ветку разработчика, симулируя новую работу). Каждый студент выполняет не менее двух изменений в каждую ветку. Возможны разные workflow (например, единый центральнй репозиторий или отдельные репозитории разработчиков; слияние веток без обеспечения линейности или использование rebase для обеспечения линейности). При этом каждый студент находится как в роли исполнителя, так и в роли руководителя для другого студента, работу в чьей ветке он ревьювит и сливает в главную. Задания на изменения в коде формулируются элементарные (например, "измени пределы цикла на строке такой-то"), но сжатое время выполнения задания (20-30 минут) используется для создания стресса и симуляции ситуации обычной разработки когда все внимание в коде и на корректное выполнение операций в системе контроля версий его остается мало. После выполнения, преподаватель разбирает с группой получившийся репозиторий и сделанные в нем ошибки.

Критерии оценивания:
корректность в роли руководителя - 0-2 балла 
корректность в роли исполнителя - 0-1 балл

Типовые ошибки:
 * коммит нарушающий синтаксис;
 * неясное сообщение коммита;
 * слияние не той ветки (чаще всего главной ветки в ветку разработчика).
